# Abyss Breaker — Project Specification

## 1. 프로젝트 개요

**Abyss Breaker**는 모바일 세로 화면에 최적화된 벽돌깨기 로그라이크 웹게임이다.

개발 목표는 기능을 많이 추가하는 것이 아니라, 짧은 조작으로 반복해서 플레이할 수 있는 코어 게임을 완성하는 것이다.

게임의 핵심 요소는 다음과 같다.

* 드래그 조준과 발사
* 공과 벽돌의 반사 충돌
* 몬스터 벽돌 전투
* 턴과 웨이브 진행
* 로그라이크 업그레이드 선택
* 정예 및 보스 전투
* 게임오버 후 영구 성장
* 모바일 한 화면 플레이

복잡한 RPG 메뉴, 인벤토리, 장비창, 다중 스크롤 UI는 만들지 않는다.

---

## 2. 개발 우선순위

기능과 디자인을 판단할 때 다음 순서를 따른다.

1. 조준과 공 반사가 재미있는가
2. 터치 조작이 정확하고 안정적인가
3. 한 화면에서 전투 상황을 바로 이해할 수 있는가
4. 반복 플레이할 이유가 있는가
5. 모바일 성능이 안정적인가
6. 시각 효과와 콘텐츠 확장

새 기능이 코어 플레이를 방해하거나 UI를 복잡하게 만들면 추가하지 않는다.

---

## 3. 핵심 플레이 흐름

```text
게임 시작
→ 화면을 드래그하여 조준
→ 손을 놓아 공 발사
→ 공이 벽과 몬스터 벽돌에 충돌
→ 벽돌 HP 감소
→ 공이 하단으로 복귀
→ 모든 공 복귀 후 턴 종료
→ 살아 있는 벽돌이 아래로 한 칸 이동
→ 플레이어 영역에 도달한 벽돌이 공격
→ 모든 벽돌을 제거하면 웨이브 클리어
→ 업그레이드 조건 충족 시 3개 중 1개 선택
→ 다음 웨이브
→ 정예 및 보스 도전
→ 플레이어 HP가 0이 되면 게임오버
→ 영구 재화 획득
→ 다음 플레이 시작
```

## 5. 작업 환경

현재 프로젝트는 VS Code와 Codex를 이용해 수정한다.

Git push나 자동 배포보다 GitHub 웹에서 파일 내용을 복사하여 반영하는 방식을 주로 사용한다.

따라서 다음 원칙을 지킨다.

* 파일 전체 교체가 가능하도록 독립적으로 작성한다.
* 일부 코드만 생략하거나 `...`으로 대체하지 않는다.
* 기존 파일을 수정하기 전에 현재 파일 전체를 확인한다.
* 기존 함수명, DOM ID, 전역 객체와의 연결 관계를 검토한다.
* 요청받지 않은 파일을 불필요하게 수정하지 않는다.
* 새 파일은 실제 복잡도를 줄일 때만 추가한다.
* 변경 후 수정한 파일 목록을 명확히 알린다.
* 브라우저 콘솔 오류가 없는지 확인한다.

## 7. 스크립트 로딩 순서

`index.html`의 `body` 종료 직전에 다음 순서로 로드한다.

```html
<script src="data.js"></script>
<script src="state.js"></script>
<script src="game.js"></script>
<script src="render.js"></script>
<script src="ui.js"></script>
<script src="main.js"></script>
```

이 순서를 변경하지 않는다.

앞쪽 파일은 뒤쪽 파일을 직접 실행 시점에 요구하지 않도록 설계한다.

모든 파일은 동일한 전역 네임스페이스를 사용한다.

```javascript
window.AbyssBreaker = window.AbyssBreaker || {};
```

권장 구조:

```text
AbyssBreaker.Data
AbyssBreaker.State
AbyssBreaker.Game
AbyssBreaker.Render
AbyssBreaker.UI
AbyssBreaker.Main
```

전역 함수와 전역 변수를 무분별하게 생성하지 않는다.

## 8. 화면 구성

전체 게임은 한 화면에 표시한다.

```text
게임 앱
 ├─ 상단 HUD
 │   ├─ 플레이어 HP
 │   ├─ 현재 웨이브
 │   ├─ 골드 또는 점수
 │   └─ 최고 웨이브
 │
 ├─ Canvas 전투 영역
 │   ├─ 몬스터 벽돌
 │   ├─ 공
 │   ├─ 조준선
 │   ├─ 파티클
 │   └─ 플레이어 위험선
 │
 ├─ 하단 컨트롤
 │   ├─ 조준 안내
 │   ├─ 일시정지
 │   └─ 재시작
 │
 ├─ 시작 오버레이
 ├─ 일시정지 오버레이
 ├─ 게임오버 오버레이
 └─ 업그레이드 선택 오버레이
```

### UI 원칙

* 전체 페이지 스크롤 금지
* 중첩 스크롤 금지
* Canvas가 화면 대부분을 차지
* HUD 높이를 최소화
* 하단 버튼 영역을 최소화
* 설명문과 카드를 과도하게 배치하지 않음
* 전투 중 필요한 정보만 표시
* 모바일 세로 화면 우선
* PC에서는 최대 너비 약 480px의 게임 화면을 중앙 정렬
* 320px 정도의 작은 모바일 화면도 대응
* 화면 노치와 홈 인디케이터를 고려
* `100dvh`를 우선 사용하고 `100vh` fallback 제공
* `safe-area-inset-top`, `safe-area-inset-bottom` 적용
* 모바일 확대, 스크롤, 텍스트 선택이 조작을 방해하지 않게 처리
* Canvas에는 적절한 `touch-action` 설정
* 버튼 터치 영역은 최소 약 44px 확보

## 10. 게임 상태

게임 모드는 명확한 상태 값으로 관리한다.

권장 모드:

```text
boot
ready
aiming
launching
playing
resolving
waveClear
upgrade
paused
gameover
```

한 번에 하나의 주요 모드만 활성화한다.

예시:

* `ready`: 조준 가능
* `aiming`: 드래그 중
* `launching`: 여러 공을 순차 발사 중
* `playing`: 공이 이동 중
* `resolving`: 모든 공 복귀 후 턴 정산
* `upgrade`: 업그레이드 선택 중
* `paused`: 게임 정지
* `gameover`: 플레이 종료

일시정지 상태에서는 물리 계산과 타이머가 진행되지 않아야 한다.

## 14. 그리드와 전투 영역

초기 권장값:

```text
열 수: 7
상단 생성 영역: 1~2행
전투 가능 행: 약 8~10행
하단: 플레이어 위험 영역
```

정확한 행 수와 간격은 Canvas 크기에 따라 계산한다.

다음 값은 데이터 또는 상수로 관리한다.

* 열 수
* 행 수
* 벽돌 간격
* 상단 여백
* 위험선 위치
* 발사선 위치
* 공 반지름
* 공 기본 속도

벽돌 크기를 여러 파일에서 별도로 계산하지 않는다.

공통 계산 함수 또는 공통 레이아웃 정보를 사용한다.

## 15. 조준과 발사

### 조준

* 발사 지점을 기준으로 드래그한다.
* 드래그 방향의 반대가 아니라 실제 발사 방향이 직관적으로 보이게 한다.
* 아래쪽 방향으로 발사할 수 없게 제한한다.
* 지나치게 수평인 각도는 최소 상승 각도로 보정한다.
* 조준 중 예상 경로 또는 점선 조준선을 표시한다.
* Canvas 밖으로 포인터가 이동해도 정상적으로 입력을 종료할 수 있어야 한다.

### 입력 방식

Pointer Events 사용을 우선한다.

지원 입력:

* 모바일 터치
* 마우스 드래그
* 스타일러스

중복 발사를 막기 위해 터치 이벤트와 마우스 이벤트를 별도로 동시에 등록하지 않는다.

### 발사

* 손을 놓으면 발사한다.
* 공이 여러 개이면 같은 각도로 일정 간격을 두고 순차 발사한다.
* 모든 공이 복귀하기 전에는 다시 조준할 수 없다.
* 발사 중 일시정지가 가능해야 한다.
* 재시작 시 발사 타이머와 예약된 발사를 모두 초기화한다.

---

## 16. 공 복귀와 다음 발사 위치

공은 하단에서 반사되지 않고 복귀 처리한다.

권장 방식:

1. 공이 하단 복귀선을 아래 방향으로 통과하면 비활성화한다.
2. 첫 번째로 복귀한 공의 X 좌표를 다음 발사 위치로 사용한다.
3. 나머지 공도 복귀하면 동일한 발사 위치에 정렬한다.
4. 모든 공이 복귀하면 턴 종료 처리를 시작한다.

발사 위치는 Canvas 좌우 경계를 벗어나지 않도록 제한한다.

공이 비정상적으로 화면 밖에 남아 턴이 끝나지 않는 경우를 대비해 안전 복구 조건을 둔다.

## 17. 턴 진행

한 번의 조준과 발사를 한 턴으로 정의한다.

```text
조준
→ 발사
→ 공 이동과 충돌
→ 모든 공 복귀
→ 턴 정산
→ 살아 있는 벽돌 한 칸 하강
→ 위험 영역 도달 판정
→ 플레이어 피해
→ 게임오버 또는 다음 턴
```

턴 정산 중에는 입력을 받지 않는다.

턴 종료 로직이 한 번만 실행되도록 상태 값을 사용한다.

## 18. 웨이브 진행

하나의 웨이브는 하나의 몬스터 벽돌 배치로 구성한다.

* 웨이브 시작 시 상단에 벽돌 배치를 생성한다.
* 여러 턴에 걸쳐 벽돌을 제거할 수 있다.
* 턴이 끝날 때마다 살아 있는 벽돌은 아래로 한 칸 이동한다.
* 모든 벽돌이 제거되면 해당 웨이브를 클리어한다.
* 웨이브 클리어 후 다음 웨이브를 생성한다.
* 업그레이드 조건을 충족하면 다음 웨이브 전에 선택 화면을 표시한다.

v0.1에서는 일반 웨이브만 구현한다.

향후 기본 구조:

```text
1~3 웨이브: 일반
4 웨이브: 정예
5 웨이브: 보스
```

정확한 주기는 `data.js`에서 변경 가능해야 한다.

---

## 19. 벽돌의 하강과 플레이어 피해

턴 종료 후 살아 있는 벽돌은 한 행 아래로 이동한다.

벽돌이 플레이어 위험 영역에 도달하면 다음과 같이 처리한다.

1. 벽돌의 공격력만큼 플레이어 HP 감소
2. 피격 효과와 화면 흔들림 발생
3. 해당 벽돌은 전투 영역에서 제거
4. 플레이어 HP가 0 이하이면 게임오버

위 방식은 v0.1 기본 규칙이다.

향후 특정 몬스터가 도달 후에도 남거나 특수 공격을 하는 구조로 확장할 수 있다.

---

## 20. 피해 계산

기본 피해 계산 순서:

```text
기본 공 피해
→ 첫 충돌 보너스
→ 치명타 판정
→ 특수 공 효과
→ 보스 피해 보정
→ 최종 피해 적용
```

v0.1에서는 기본 피해와 치명타만 사용해도 된다.

피해 값은 최소 1 이상이 되도록 한다.

벽돌 HP가 0 이하가 되면 즉시 파괴 상태로 처리한다.

동일한 벽돌의 파괴 보상은 한 번만 지급한다.

---

## 21. 충돌 처리

### 기본 방식

* 공은 원으로 처리
* 벽돌은 축 정렬 사각형으로 처리
* 원과 사각형의 최근접점 방식으로 충돌 판정
* 충돌 법선을 기준으로 공 속도를 반사
* 충돌 후 공을 벽돌 바깥으로 이동시켜 내부 고착 방지

### 벽 충돌

* 좌측 벽: X 속도 반전
* 우측 벽: X 속도 반전
* 상단 벽: Y 속도 반전
* 하단: 반사하지 않고 복귀 처리

충돌 후 공 위치를 벽 내부에 남겨두지 않는다.

### 빠른 공 대응

공 속도가 빨라져도 벽돌을 통과하지 않도록 한다.

권장 방식:

* 한 프레임의 이동 거리를 여러 서브 스텝으로 나눔
* 공 반지름이나 최대 이동 거리를 기준으로 서브 스텝 수 결정
* 서브 스텝 최대 횟수를 제한해 성능 보호

### 중복 충돌 방지

* 한 프레임 또는 짧은 시간 안에 같은 벽돌에 피해가 중복 적용되지 않게 한다.
* 공별 충돌 기록 또는 짧은 충돌 쿨다운을 사용한다.
* 관통 효과가 없는 공은 충돌 후 정상적으로 반사한다.
* 파괴된 벽돌은 이후 충돌 대상에서 즉시 제외한다.

## 22. 시간 처리

게임 물리는 delta time 기반으로 처리한다.

```javascript
deltaTime = Math.min(deltaTime, MAX_DELTA_TIME);
```

브라우저가 잠시 비활성화되거나 프레임이 멈춘 후 공이 순간 이동하지 않도록 delta time 상한을 둔다.

권장 상한:

```text
약 0.033초~0.05초
```

브라우저 탭이 다시 활성화될 때 이전 프레임 시간을 초기화한다.

## 23. Canvas 크기와 해상도

CSS 표시 크기와 Canvas 내부 해상도를 분리한다.

* CSS에서는 Canvas가 컨테이너를 채움
* JavaScript에서 실제 표시 크기를 읽음
* devicePixelRatio를 고려해 내부 해상도 설정
* 지나치게 높은 DPR은 성능을 위해 제한 가능
* 리사이즈 후 좌표계와 벽돌 배치를 안정적으로 갱신
* 화면 회전 후 게임이 깨지지 않아야 함

물리 좌표와 렌더링 좌표가 서로 다른 기준을 사용하지 않도록 한다.

가능하면 논리 좌표계를 하나로 통일한다.

권장 구조:

## 32. 코드 작성 원칙

* 각 파일 최상단에 필요하면 `"use strict";` 사용
* 의미 없는 전역 변수 생성 금지
* 매직 넘버 최소화
* 주요 수치는 `data.js`의 상수로 관리
* 함수 하나가 지나치게 많은 책임을 갖지 않게 분리
* 파일 간 순환 의존성 방지
* 존재하지 않는 함수 호출 금지
* 존재하지 않는 DOM ID 참조 금지
* NaN이 물리 좌표에 들어가지 않게 방어
* `Infinity`와 비정상 좌표 검사
* 빈 배열과 누락된 데이터 처리
* 같은 턴 종료 로직이 중복 실행되지 않게 처리
* 같은 보상이 중복 지급되지 않게 처리
* 이벤트 리스너가 재시작할 때 중복 등록되지 않게 처리
* `requestAnimationFrame` 루프를 여러 번 시작하지 않음
* 재시작 시 타이머와 입력 상태 초기화
* 일시정지 해제 시 delta time 급증 방지
* 주석은 코드 설명이 필요한 부분에만 사용
* 필요 이상으로 추상화하거나 클래스를 늘리지 않음

## 33. 오류 방지 체크

변경 후 다음 항목을 확인한다.

### 초기화

* 모든 스크립트가 순서대로 로드되는가
* `AbyssBreaker` 네임스페이스가 정상 생성되는가
* Canvas와 주요 DOM 요소가 존재하는가
* 게임 루프가 한 번만 시작되는가

### 입력

* 모바일 드래그가 페이지 스크롤을 발생시키지 않는가
* 마우스와 터치가 중복 발사를 일으키지 않는가
* Canvas 밖에서 손을 떼도 입력이 종료되는가
* 발사 중 추가 입력이 차단되는가

### 물리

* 공 좌표에 NaN이 발생하지 않는가
* 공이 벽돌 내부에 갇히지 않는가
* 공이 빠를 때 벽돌을 통과하지 않는가
* 동일 충돌이 한 프레임에 중복 피해를 주지 않는가
* 공이 화면 밖에서 영구적으로 남지 않는가
* 모든 공 복귀 후 턴이 정확히 한 번 종료되는가

### 상태

* 일시정지 중 게임이 진행되지 않는가
* 재시작 후 이전 공과 벽돌이 남지 않는가
* 게임오버 후 조준할 수 없는가
* 새 게임에서 HP와 웨이브가 정상 초기화되는가

### UI

* 페이지 스크롤이 생기지 않는가
* 작은 화면에서 버튼이 겹치지 않는가
* Canvas가 화면 대부분을 차지하는가
* 오버레이가 숨겨졌을 때 클릭을 막지 않는가
* PC에서 게임 화면이 중앙 정렬되는가

### 저장

* 최고 웨이브가 저장되는가
* 새로고침 후 최고 웨이브가 유지되는가
* 저장 데이터가 손상되어도 기본 상태로 실행되는가

```text
→ PROJECT_SPEC.md
→ 현재 정상 작동 중인 코드 구조
→ Codex의 일반적인 판단
```

기존 정상 기능을 제거하거나 구조를 크게 변경해야 한다면 그 이유를 결과에 명시한다.

---

## 35. Codex 결과 보고 형식

작업 완료 후 다음 내용을 간단히 보고한다.

```text
1. 작업 요약
2. 수정한 파일
3. 구현한 기능
4. 테스트 방법
5. 확인이 필요한 사항
6. 추천 Commit 메시지
```

채팅에서 전체 코드를 요청받은 경우:

* 파일 전체 코드를 제공
* 일부 코드 생략 금지
* 변경되지 않은 부분도 포함
* 코드 블록 상단에 파일명 표시
* 다음 단계 요청 문장 제공

워크스페이스에서 직접 파일을 수정한 경우:

* 실제 파일을 수정
* 채팅에 모든 코드를 반복 출력하지 않음
* 수정한 파일과 핵심 변경 사항을 보고
* 실행 및 테스트 방법을 제공

## 36. 완료 기준

기능은 코드가 존재하는 것만으로 완료된 것으로 보지 않는다.

다음 조건을 충족해야 완료로 판단한다.

* 요청한 기능이 실제 플레이 흐름에 연결됨
* 모바일 터치로 조작 가능
* PC 마우스로 조작 가능
* 콘솔 오류 없음
* 게임 루프가 중복 실행되지 않음
* 재시작 가능
* 일시정지 가능
* 화면 스크롤과 UI 겹침 없음
* 파일 간 참조 오류 없음
* 기존 기능이 유지됨
* 테스트 방법이 제공됨
